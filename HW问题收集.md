1. 如何让领域对象真正接入 Svelte 的界面流程？

   1. **上下文**：一开始 `Sudoku` 和 `Game` 虽然在测试中可以工作，但前端界面仍然主要直接依赖原来的 store 和二维数组，导致领域对象没有真正成为运行时核心。
   2. **尝试解决手段**：查阅了 Svelte store 的使用方式，并结合项目现有结构增加了 `gameStore` 作为 adapter，让界面统一通过 `gameStore` 读取状态、调用 `guess / undo / redo / startNew` 等方法，再由 adapter 去驱动 `Game` 和 `Sudoku`。

2. 如何稳定实现 undo / redo，而不是只在局部逻辑里临时回滚？

   1. **上下文**：原项目里撤销重做并没有完整实现，如果直接记录对数组的引用，后续修改会污染历史状态，导致回退结果不可靠。
   2. **尝试解决手段**：通过阅读测试和自己补充理解对象拷贝、序列化的区别，最终改成保存 `Sudoku` snapshot 的方式管理历史记录，并在每次操作后同步 `canUndo` 和 `canRedo`，让撤销重做有明确的状态边界。

3. 如何判断当前棋盘中的冲突格，并把结果稳定暴露给 UI？

   1. **上下文**：如果把冲突判断散落在组件里，逻辑会重复，而且容易出现“数据变了但某些组件没有同步更新”的问题。
   2. **尝试解决手段**：自己梳理了数独行、列、宫的约束规则，把冲突检测收拢到领域对象和 adapter 的同步流程里，对外统一暴露 `invalidCells`，让界面只负责消费结果和渲染高亮。

4. 如何进一步划清领域状态与 UI/session 状态的边界？

   1. **上下文**：当前已经把 `Sudoku`、`Game`、undo/redo 等核心逻辑放进领域层，但像 `notes`、`cursor`、`hints`、暂停状态等内容仍然主要停留在前端 store 中。
   2. **尝试解决手段**：目前主要通过阅读现有代码、分析不同状态的职责来拆分；我已经能区分哪些属于业务规则，哪些更偏界面交互，但还没有形成一套完全统一、可复用的边界划分方案。

5. 如果后续继续扩展功能，怎样避免 adapter 层变得越来越臃肿？

   1. **上下文**：现在 `gameStore` 负责把领域对象转换成 view model，同时还承担部分操作入口；随着功能继续增加，这一层可能会逐渐堆积过多职责。
   2. **尝试解决手段**：目前通过控制对外暴露的数据结构、保持接口集中来减轻复杂度，也参考了分层设计的思路；但对于未来是否要继续拆分更多 service / mapper / action 层，我还没有完全想清楚。

6. 如何在保证设计清晰的同时，兼顾性能与实现成本？

   1. **上下文**：当前方案中会把领域状态重新投影成 view model，undo/redo 也采用 snapshot 方式，这种做法结构清晰，但会带来一定的数据复制和内存开销。
   2. **尝试解决手段**：现阶段优先保证作业要求中的领域建模清晰、响应式边界明确，并暂时接受这部分 trade-off；但如果后续题盘规模、功能复杂度继续提升，怎样在不破坏现有设计的情况下做更细致的性能优化，仍然是一个待解决的问题。
