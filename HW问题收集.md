### 已解决

1. 啥是“标量 derived store”，有啥作用？
   1. **上下文**：在整理这次作业设计时，出现了“UI 的棋盘、输入、Undo/Redo、胜利判断、分享编码全部直接读取领域对象公开接口；允许保留少量标量 derived store，但不再生成板级 view model”这句话。当时我不太清楚“标量 derived store”具体指什么，也不知道它为什么还能保留。
   2. **解决手段**：通过直接询问 Coding Agent，并结合 Svelte store 文档和项目代码理解，确认这里的意思是像 `gameWon`、`keyboardDisabled` 这种由已有 store 推导出来的单个布尔值或简单值。它的作用是保留轻量的响应式组合，但不再把整块棋盘重新包装成一个额外的大型 view model。

2. 为什么这次要引入 `gameStore`，而不是继续让组件直接改二维数组？
   1. **上下文**：一开始我对“领域对象接入真实前端流程”这件事理解不深，感觉组件直接改数据也能跑，不太明白为什么还要多一层 adapter store。
   2. **解决手段**：通过阅读 `src/node_modules/@sudoku/stores/game.js` 和界面组件调用链，我逐渐理解了 `gameStore` 的作用：它一方面统一暴露 `guess / undo / redo / startNew` 等操作入口，另一方面把 `Game` 和 `Sudoku` 的状态同步成界面可直接消费的数据，从而把 UI 和领域逻辑之间的边界稳定下来。

3. undo / redo 为什么不能只存数组引用，为什么要存 snapshot？
   1. **上下文**：在实现撤销重做时，我最开始对“历史状态到底该怎么存”没有把握，直觉上觉得把当前棋盘直接塞进栈里就行。
   2. **解决手段**：通过看测试、读 `src/domain/game.js`，再自己梳理对象引用和拷贝的区别，我意识到如果只保存同一个数组的引用，后面的修改会污染前面的历史记录。最后理解了用 `Sudoku` snapshot 保存状态的做法，这样 undo / redo 回退的才是真正独立的历史快照。

### 未解决

1. 这个 `sameArea` 有啥用啊？
   1. **上下文**：`src/components/Board/index.svelte`

      ```javascript
      sameArea={$settings.highlightCells && !isSelected($cursor, x, y) && isSameArea($cursor, x, y)}
      ```

   2. **尝试解决手段**：我已经顺着代码读到 `isSameArea()` 的实现，知道它会判断某个格子是否和当前选中格子处在同行、同列或同一个九宫格里，也知道这个值会传给 `Cell.svelte` 控制高亮样式；但我还没完全想清楚这个命名是否过于宽泛，以及它和 `sameNumber`、`conflictingNumber` 这几类高亮在交互层面应该怎样更清晰地区分。

2. 哪些状态应该放进领域层，哪些状态应该继续留在 UI/store 层？
   1. **上下文**：现在 `Sudoku`、`Game`、`undo/redo`、冲突检测这些核心规则已经比较明确，但像 `cursor`、`notes`、`hints`、暂停状态等内容，仍然分散在多个前端 store 中。
   2. **尝试解决手段**：我目前通过阅读现有实现和设计文档，能大概区分“业务规则”和“界面交互状态”，但对于某些边界情况还不够自信。例如提示次数、候选数、暂停状态究竟算游戏规则的一部分，还是只是前端会话状态，我还没有形成一套完全稳定的判断标准。

3. 后续如果功能继续加，`gameStore` 会不会越来越臃肿？
   1. **上下文**：现在 `gameStore` 既承担了领域对象到界面数据的同步，也承担了不少操作入口；当前规模还可以接受，但我担心后面功能变多后，这一层会逐渐变成“什么都往里放”的位置。
   2. **尝试解决手段**：我已经尝试从分层设计角度去理解 adapter 的职责，也看了当前哪些内容是同步 view state，哪些内容是操作封装；但如果以后要继续拆成 mapper、service 或 action 层，应该拆到什么程度、以什么标准拆，我还没有完全想清楚。
